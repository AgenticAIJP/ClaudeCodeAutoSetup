# style.md の使い方(人間向け解説)

> 隣の `style.md` は Claude が従うコーディング規約です。
> チームの規約に合わせて自由に書き換えてください。

## なぜ必要か

規約がないと、Claude はセッションごとに微妙に違うスタイルで書きます。
命名・エラー処理・コメント言語がバラバラになり、レビューコストが増えます。

## Before / After

### ✅ style.md がある場合

→ 全セッションで一貫したスタイル。「コメントは英語で」と毎回言わなくてよい
→ 「バグ修正の前にテストを書く」ルールにより、修正の質が安定する

### ❌ ない場合

→ ある日は日本語コメント、ある日は英語。関数の長さも気分次第
→ エラー処理が `catch {}` で握りつぶされていても気づかない

## カスタマイズ例

```markdown
- インデントはスペース2つ
- React コンポーネントは関数コンポーネントのみ。class 禁止
- CSS は Tailwind のユーティリティクラスのみ使用
- console.log をコミットに含めない
```

## コツ

linter で機械的にチェックできるものは linter に任せ、
ここには **linter では表現できない判断基準** を書くのが効率的です。
