# architecture.md の使い方(人間向け解説)

> 隣の `architecture.md` は Claude が読む「アーキテクチャの憲法」です。
> CLAUDE.md から `@.claude/rules/architecture.md` で読み込まれています。

## なぜ必要か

Claude は指示がなければ**最短経路**でコードを書きます。
それは速いが、層を無視した「なんでも main に書く」コードになりがちです。

## Before / After

### ✅ architecture.md がある場合

ユーザー「決済機能を追加して」
→ Claude は `src/core/payment.ts` を作り、エントリポイントからは呼び出すだけにする
→ 依存方向が保たれ、テストしやすい構造が維持される

### ❌ ない場合

ユーザー「決済機能を追加して」
→ Claude は `main.ts` に決済ロジックを直接書く(最短経路)
→ 3回の機能追加で main.ts が 500 行になり、アーキテクチャが崩壊する

## 書き方のコツ

- **短く、断定形で**。長い説明は効果が薄れる(Claude のコンテキストを消費する)
- 「〜を推奨」ではなく「〜する / 〜しない」と書く
- プロジェクト固有の禁止事項(触ってはいけないファイル等)を必ず入れる

## カスタマイズ例

```markdown
- `src/legacy/` は変更禁止。読み取りのみ
- DB アクセスは必ず Repository クラス経由
- 環境変数は config.ts でのみ読む。直接 process.env を参照しない
```
